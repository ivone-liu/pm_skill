# Diagram Playbook

## Use flowchart when
You need to show the main user path, branching, and outcome.

## Use swimlane when
Multiple roles or systems interact and handoffs matter.

## Use state machine when
An object changes status and available actions differ by state.

## Use wireframe when
A single page has dense modules and structure clarity matters more than visual polish.

## Use architecture diagram when
The user journey depends on service boundaries, async workers, storage, or API interactions.

## Use timeline when
The request includes phased rollout, milestones, or coordinated delivery rhythm.

## Rule of thumb
A diagram should answer one ambiguity clearly. If it does not reduce disagreement, it is decorative noise.
