# PRD Writing Principles

## 1. A PRD is not a pretty note
A usable PRD aligns product, design, engineering, QA, and operations around the same implementation target.

## 2. Good structure is adaptive
The section order can change, but the document must still answer:
- why this exists
- who it serves
- what changes
- how it works
- what can fail
- how to verify it

## 3. Separate facts from assumptions
If something is inferred, label it. Hidden assumptions are one of the main sources of downstream rework.

## 4. Describe mechanisms, not slogans
“提升体验” is not a mechanism. State what changes, for whom, and through what interaction or logic.

## 5. A good PRD reduces guessing
If design, engineering, or QA still need to guess the main behavior, the PRD is not done.
